# Source Ledger

This ledger records where rules, rulesets, and tools come from. Update it whenever the shared config, custom rule surface, policy scripts, hooks, Sonar profile, or consumer-tooling guidance changes.

The goal is provenance, not duplicate rule documentation. Detailed status still lives in:

- `policy/registries/rules.yaml` for custom rule maturity, examples, external overlap, and promotion state.
- `docs/rule-status-registry.md` for the readable custom-rule index.
- `docs/lint-rule-parity.md` for retired Oxlint parity.
- `docs/rule-equivalence-audit.md` for custom-versus-ecosystem decisions.
- `docs/sonarqube.md` for Sonar quality profile and external issue import guidance.

## Source Types

| Source type          | Meaning                                                                                             | Required note when adding                                                        |
| -------------------- | --------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| `local-original`     | Implemented in `@joedeleeuw/antidrift` because no maintained equivalent covers the scoped behavior. | Link the rule registry row, investigation doc, and real-corpus gate.             |
| `ecosystem-ruleset`  | Pulled from a maintained ESLint plugin or config.                                                   | Name the package, exact rule/ruleset, and why it is used instead of custom code. |
| `generated-config`   | Expressed with core ESLint config generated from policy registries.                                 | Name the registry and generated core rule.                                       |
| `delegated-tool`     | Owned by Sonar, hooks, TypeScript, Vitest, or another non-ESLint gate.                              | Name the tool and local invocation/config path.                                  |
| `consumer-tooling`   | Recommended for consuming repos, not bundled into antidrift.                                        | Name the consumer source and why it stays outside the package.                   |
| `borrowed-reference` | Adopted after reviewing another local repo, but enforced through maintained rules or local policy.  | Name the reference repo/path and the final owning rule/tool.                     |

## Addition Checklist

Before adding a rule, ruleset, or tool:

1. Pick one source type from this file.
2. Record the package/tool/rule IDs and local config path.
3. If it is custom, update `policy/registries/rules.yaml` and the relevant investigation/audit docs.
4. If it replaces custom code or a retired engine rule, update `docs/lint-rule-parity.md` or `docs/rule-equivalence-audit.md`.
5. If it came from reviewing another repo such as Chaski, record that path as a `borrowed-reference` and name the final owner.
6. If it is consumer-only tooling, document why it is not bundled into antidrift.
