# Semantic Validation Matrix

This matrix turns the semantic drift goal into an execution checklist. Use it before adding, promoting, retiring, or migrating an `antidrift/*` rule.

The rule name is not the proof. A rule blocks only when its carrier can prove the broken association it claims to own. When the carrier cannot prove that association, the rule stays inventory-only, research, delegated to ecosystem tooling, or retired.

## How To Use This Matrix

Every active custom rule and serious candidate needs a row before promotion. The row must answer:

- What proof object justifies the finding?
- Which carrier owns the proof: plain ESLint, type-aware ESLint, semantic adapter, authority registry, repo graph, agent-ops, or model-assisted discovery?
- What exact threshold turns evidence into blocking behavior?
- Which real corpus files prove drift and clean controls?
- What happens when the semantic fact sink, registry, parser services, or corpus evidence is missing?

Reduced fixtures are regression locks only. Real codebase copies are the promotion gate for drift, clean controls, owner discovery, and false-positive pressure.

## Validation Requirements By Carrier

| Carrier                        | Required proof before promotion                                                                                                                                                        |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Plain AST or class-string rule | RuleTester coverage, real drift file, real clean control, ecosystem comparison, and documented false-positive boundary.                                                                |
| Type-aware ESLint rule         | Parser-services failure behavior, RuleTester coverage with project service, real type-aware drift and clean controls, and a non-type-aware story if the rule can run without services. |
| Semantic adapter               | Adapter unit tests, rule integration tests, fact emission where useful, no-sink behavior, invalidation cases, and real corpus classification.                                          |
| Authority registry             | Registry validation, owner-file clean controls, non-owner drift, missing-registry behavior, generated config or typed config smoke, and no unaccepted dependency-wide blocking.        |
| Repo graph or policy script    | Structured input validation, changed-file or graph evidence, replayable command output, and protected-path clean controls.                                                             |
| Agent-ops                      | Transcript or hook replay tests, timestamp or command-fingerprint freshness, adapter coverage for Codex, Claude, OpenCode, and PR surfaces where claimed.                              |
| Model-assisted exploration     | Offline evaluation against real corpus, precision or recall report, deterministic downstream review path, and no blocking dependency.                                                  |

## Success Criteria

Code structure is successful when rules share adapters instead of each embedding private semantic recovery, public config types expose the required settings, and package consumers can enable the rules without reaching into repo internals.

Semantic fact structure is successful when every fact kind emitted by a rule is declared in `policy/registries/rules.yaml` with its owning rules, adapter ID, carrier, confidence, emission mode, semantic association, no-sink behavior, and payload fields. Fact kinds are part of the package contract, not throwaway debugging output.

Data accuracy is successful when each blocking branch has real-corpus drift, real clean controls, no-sink or missing-config behavior, parser-service behavior where relevant, and an ecosystem comparison that explains why custom code remains necessary.

Completion is successful when every active rule has a registry row, matrix row, RuleTester coverage, corpus evidence, documented severity, and a package verification path. Rules that cannot meet the semantic proof bar must be default-off, inventory-only, ecosystem-delegated, agent-ops-only, research, or retired.
