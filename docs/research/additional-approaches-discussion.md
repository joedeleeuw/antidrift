# Additional technical approaches — discussion

## 1. diff-scope-creep (n=60)

The additional 2026 material strengthens the diagnosis, but not the proof story. Agentic review research now explicitly treats code review as a multi-stage, context-carrying workflow with specialized agents and human quality gates ([arXiv:2605.17548](https://arxiv.org/abs/2605.17548)). PR-message/code inconsistency work shows the artifact is worth checking: 23,247 agentic PRs yielded measurable PR-description mismatch, with high-inconsistency PRs taking longer and being accepted far less often ([arXiv:2601.04886](https://arxiv.org/abs/2601.04886)). But that is still description-vs-diff, not task-vs-diff. SWE-PRBench is the warning label: frontier LLM reviewers detected only 15-31% of human-flagged issues in diff-only mode, and more context degraded performance ([arXiv:2603.26130](https://arxiv.org/abs/2603.26130)).

The deterministic core we could own is not “prove the natural-language task was understood.” That remains human/LLM territory. The deterministic core is narrower and defensible: **prove conformance to a machine-readable change contract**. If an intent artifact declares allowed files, owner symbols, public APIs, dependency surfaces, operation classes, and negative scope, then a diff-relative spine can prove violations: changed path outside declared scope, new exported symbol not declared, new route/schema/service owner, new dependency, touched graph radius outside allowed entrypoints, or public surface expansion without a matching contract entry. That is deterministic if the artifact exists before editing and the diff is compared against merge-base.

The probabilistic layer sits before and beside that core. LLMs can infer a draft change contract from Jira/ADO, issue text, or PR description, and can flag “this manifest seems too broad for the ticket.” They can also summarize suspicious structural facts. They cannot be the blocker. Under the stance in `docs/whitepaper-agent-code-drift-proofs.md` and Learning 10 in `docs/research-learnings.md`, the blocking claim should be “diff violates declared machine-readable scope,” not “LLM thinks this exceeded intent.”

Market gap: I do not see a deterministic market proof for true scope-vs-intent. Drift’s inventory says its change-aware layer covers churn, coupling, recent imports, and blast-radius notices, but not stated intent conformance (`docs/research/drift-rule-inventory.md`). Vendor AI review and ticket-aware tools are useful process controls, not deterministic proof. The novelty is turning intent into a **proof-carrying change contract** and checking structural diff conformance through the existing fact/registry spine.

Fit is strong. `policy/registries/solve-buckets.yaml` already labels `diff-scope-creep` as needing `diff-relative`, and the architecture review says a change context is the missing seam while the `semanticFact` sink is the right aggregation point. A new fact could carry `declaredScope`, `actualChangeSurface`, `violations`, and `confidence: deterministic-enforcement` only for contract breaches. Evidence-gating should start from the n=60 complaint corpus, with multi-repo clean controls for legitimate broad refactors. Effort: medium for manifest + file/symbol/API/dependency checks; high for refactoring-aware TypeScript AST diff.

## 2. net-new-bias / owner-discovery (n=25)

Hybrid semantic code search is relevant, but only as retrieval. RepoRift’s agentic RAG approach reports 78.2% Success@10 on CodeSearchNet ([arXiv:2408.11058](https://arxiv.org/abs/2408.11058)), which is good evidence that learned retrieval can surface likely existing code. It is not proof that a new owner is a duplicate.

The clean deterministic core is a **repo-wide owner index** built from TypeScript compiler facts and structural signatures: exported types, schemas, services, route/action handlers, file-path/domain tokens, normalized symbol names, public method sets, property fingerprints, generated/domain registry relationships, and import graph neighborhoods. For every newly added public owner-like surface in the diff, the engine asks: does an existing owner have the same normalized concept tokens plus the same or near-same public contract? Does a new type structurally match an existing exported type or generated schema? Does a new service expose the same verb-object surface as an existing service in the same domain?

The boundary is straightforward. Lexical/symbol/structural matches are deterministic candidate proofs. Embeddings and LLM search are candidate generators and triage: “look at these ten possible owners before accepting the new one.” Blocking should require deterministic confirmation, such as exact or high-threshold structural type equivalence, same generated/domain authority, same exported operation set, or an existing registry owner with a provable structural match. Conceptual “auth helper resembles auth service” without structural confirmation stays inventory.

Market gap: most code search uses this for generation context, not governance. Drift’s inventory confirms it enforces configured architecture and declared owners, but does not discover new duplicate owners. Antidrift’s own Learning 7 says the authority-index can only enforce declared owners; it cannot discover net-new duplicate owners. That leaves a real product gap: **duplicate-on-write owner discovery** for TypeScript.

Fit is high but promotion risk is non-trivial. This should extend the existing `structuralMatch`/authority-registry model into a discovered-owner state rather than invent a parallel surface. Evidence gating needs clean DTO/view-model/facade controls, because many legitimate boundary contracts intentionally resemble domain models. Effort: medium for type-declaration and exported-symbol indexing; medium-high for service/route owner discovery; high if conceptual embeddings are made production-grade, which they should not be for blocking.

## 3. silent-fallback (n=24)

The extra exception-flow research is directly on point. Exception anti-pattern work used automated exception-flow analysis across Java/C# subjects ([arXiv:1704.00778](https://arxiv.org/abs/1704.00778)); later work analyzed over 10K handling blocks and 77K exception flows, showing many exceptions trace across method boundaries ([arXiv:1708.00817](https://arxiv.org/abs/1708.00817)). Parichayana is older Java tooling, but it proves the class of deterministic exception-handling smell detection is established ([arXiv:1701.00108](https://arxiv.org/abs/1701.00108)).

The deterministic core should be split by maturity. First, intra-procedural: detect catch clauses where the caught binding or error condition reaches no approved sink, no rethrow, no typed result/error return, and the body returns an empty/default value or writes fallback state. That is deterministic and likely valuable, but bounded. Second, inter-procedural: model catch-param/error-state as a source and approved sinks as logging/reporting/telemetry/rethrow/result propagation. That belongs in a dataflow/taint spine, preferably delegated to CodeQL/Joern/Semgrep rather than hand-rolled, consistent with `policy/registries/solve-buckets.yaml`.

The probabilistic layer is sink inference and acceptability judgment. LLMs can propose “this project’s `captureException`, `logger.error`, `return err(...)`, and `Result.err` are sinks,” and can explain why a fallback may be acceptable. The blocking query should only use registered sinks and deterministic reachability. “This fallback is semantically okay” is not machine-checkable unless the project encodes that policy.

Market gap: drift’s `broad_exception_monoculture` checks module-wide uniformity, not per-instance flow. Generic lint catches empty catches and console patterns; it does not prove caught error reaches a real sink. Antidrift’s gap framing is correct: `silent-fallback` is not an AST-shape rule once helper calls enter the path.

Fit is strong because the fact/registry model already distinguishes inventory from blocking. A `caughtErrorFlow` fact can carry source, fallback sink, approved error sink, rethrow/result propagation, and boundedness. Evidence-gating should use real catch-block corpora and preserve Learning 2 from `docs/research-learnings.md`: real handlers often store derived error codes/messages, not raw caught errors. Effort: medium for intra-procedural TS; high for inter-procedural unless delegated.

## 4. duplication / one-owner-fork (n=31 + n=28)

The learned clone-detection frontier is impressive but should not become a blocker. MAGNET combines AST, CFG, and DFG graphs with attention and reports strong BigCloneBench/Code Jam F1 ([arXiv:2510.24241](https://arxiv.org/abs/2510.24241)). Code-LLM concept-layer embeddings are also being used for clone detection ([arXiv:2512.19980](https://arxiv.org/abs/2512.19980)). These are useful inventory engines for Type-4 semantic clones. They are learned, benchmark-sensitive, and not deterministic proof.

The deterministic core we should own first is simpler and likely higher yield: **TypeScript declaration clone detection**. Normalize object/interface/type declarations through the TypeChecker: property names, optionality, readonly, literal unions, nullability, array/container shape, generic instantiation, schema-derived provenance, and owner registry identity. Exact structural equality and high-threshold near equality are deterministic. This directly covers the gap drift misses: its `mutant_duplicate` targets function bodies, not type declarations, and `docs/research/drift-rule-inventory.md` calls that out.

For function/body duplication, deterministic Type-2/Type-3 methods still matter: normalized AST fingerprints, token winnowing, function signature similarity, and PDG-ish local dataflow summaries. The learned layer can propose Type-4 candidates; the deterministic confirmer decides whether there is a structural fork, common owner, or exact behavioral wrapper. Blocking on learned semantic clone alone would violate the project stance.

Market gap: clone tools answer “are there clones?” Drift answers “are function bodies similar?” Antidrift’s agent-specific question is narrower: **did this diff introduce a duplicate of an existing owner, including a type declaration owner?** That diff + owner-registry framing is still open.

Fit is excellent. This is a natural extension of `structuralMatch` and the type-contract authority family. Evidence-gating should separate type declaration forks, function body clones, DTO/wire-boundary overlaps, and intentional facades. Effort: low-medium for declaration fingerprints; medium for diff-scoped cross-file clone detection; high for learned Type-4 triage infrastructure.

## Cross-cutting conclusions

Highest value-to-effort: **machine-readable change-contract conformance for diff-scope-creep**. It targets the largest single uncovered behavior (n=60), exploits the existing fact/registry backbone, and creates a deterministic claim the market does not appear to make. The key discipline is to name the claim correctly: not “proved the task intent,” but “proved the diff exceeded the declared scope contract.” A close second, and probably the fastest stable win, is deterministic type-declaration clone detection because it covers duplication + one-owner-fork and directly exploits a drift blind spot.

Dead ends and traps: LLM multi-agent review as a blocking gate; vector/embedding owner discovery as proof; Type-4 clone models as enforcement without structural confirmation; line-count/churn thresholds as scope-creep proof; incomplete call-graph “no sink exists” claims without a boundedness label; and synthetic fixtures for promotion. These can all support triage or inventory, but they should not carry `blocking-diagnostic` maturity.

One additional approach not listed: **change-impact witnesses**. For a declared task entrypoint or target test/smoke, collect deterministic static/dynamic impact evidence: import/dependency slice, V8/Jest coverage, changed exported symbols exercised by relevant tests, and unwitnessed changed surfaces. This borrows from regression-test selection, incremental coverage analysis ([arXiv:2410.21798](https://arxiv.org/abs/2410.21798)), slice-based change impact ([arXiv:2508.19056](https://arxiv.org/abs/2508.19056)), and TypeScript dynamic slicing work ([arXiv:2502.20533](https://arxiv.org/abs/2502.20533)). It still does not prove intent, but it deterministically identifies changed surfaces with no executable witness for the claimed task. That is a strong inventory signal and, for high-risk public API/owner changes, a possible future gate after multi-repo evidence.
